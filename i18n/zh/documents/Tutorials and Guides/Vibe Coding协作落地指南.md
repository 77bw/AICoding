# Vibe Coding 协作落地指南

适用于**新建项目**和**存量项目接入 AI 协作**的落地手册，帮助你把仓库里的提示词、模板和方法论真正用起来，避免失控开发。

## 1. 仓库资源速览（知道用什么）
- 系统提示词：`i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md`（通用规范，v8 亦可）。
- 任务提示词：`i18n/zh/prompts/coding_prompts/`，常用三件套：
  - `项目上下文文档生成.md`：生成/更新项目上下文。
  - `智能需求理解与研发导航引擎.md`：把模糊需求拆解成可执行方向。
  - `标准化流程.md`：将任务固化成可执行步骤。
- 方法论与最佳实践：`i18n/zh/documents/Methodology and Principles/`（开发经验、系统提示词构建原则等）。
- 模板与资源：`i18n/zh/documents/Templates and Resources/`（通用项目架构模板、代码组织、工具集等）。

## 2. 基础准备（开场就固定上下文）
1) 选模型与接口：首选 Codex/Claude 终端或 IDE 插件（支持读取文件、查看 diff）。  
2) 建“memory bank”文件：
   - `CLAUDE.md` / `AGENTS.md`：放行为准则与必须阅读的上下文。
   - `architecture.md`：记录各文件职责与数据流。
   - `progress.md`：记录完成的里程碑与已跑过的测试。
   - 需求/设计/技术栈文档（PRD/GDD、tech-stack.md、implementation-plan.md）。
3) 项目骨架：按 `通用项目架构模板.md` 选择合适的目录结构（Python Web、数据科学、Monorepo 或全栈）。
4) Git 与环境：初始化仓库、`.gitignore`、虚拟环境/lockfile，首个 commit 只包含骨架与文档。

## 3. 新项目落地步骤
1) 固化需求与技术栈  
   - 写最小可行的 PRD/技术栈文档；如果缺少架构草图，先用 `智能需求理解与研发导航引擎.md` 生成思路与候选方案。
2) 生成项目上下文文档  
   - 把已有需求/技术栈/约束喂给 `项目上下文文档生成.md`，得到首版上下文；缺信息的字段保持“暂无信息”，后续补全。
3) 配置系统提示词  
   - 复制 `system_prompts`（建议 v10）到 IDE/CLI 的系统提示，并添加 Always 规则：写代码前必须阅读 `architecture.md`、`progress.md` 和上下文文档。
4) 形成实施计划  
   - 用 `标准化流程.md` 或你偏好的计划提示词，把需求拆为小步，每步带可验证的测试/验收标准。
5) 开发循环（Ask → Plan → Execute）  
   - 每次只做一小步，先让 AI 给计划/Ask，再执行；完成后更新 `progress.md` 与 `architecture.md`，再开始下一步。
6) 交付与复盘  
   - 跑测试、整理变更摘要，必要时用 `系统架构可视化` 等提示词生成 mermaid/图表（可选）。提交小步 commit。

## 4. 老项目接入 AI 的步骤
1) 盘点与快照  
   - 现有 README/架构文档/关键目录列表，必要时用现有脚本做备份；明确非目标范围（暂不改动的模块）。
2) 补齐上下文  
   - 运行 `项目上下文文档生成.md`，输入当前仓库信息、主要模块、数据流、依赖；缺失部分保持“暂无信息”待补。
   - 整理 `architecture.md`：为核心文件写一句话职责和上下游依赖。
3) 建系统约束  
   - 同步 `system_prompts` 到 IDE，并设为 Always 读取 `architecture.md`、`progress.md`、需求与限制。
4) 渐进式改造  
   - 用 `智能需求理解与研发导航引擎.md` 澄清目标，再用 `标准化流程.md` 将改动拆小步；每步完成就更新 `progress.md`，防止上下文漂移。
5) 清理与提质  
   - 对照 `代码组织.md` 与 `开发经验.md` 统一命名、目录与注释；补上 `.gitignore`、测试与基础文档。

## 5. AI 协作守则（摘自仓库“道/法/术”）
- 目的主导、上下文先行：写代码前先让 AI 复述目标与非目标。
- 一次只改一件事：避免巨型变更，方便回滚与审阅。
- 能抄不写、能文档就文档：先查官方/现有实现，文档即上下文，不是事后补。
- 输入-处理-输出要清楚：接口先行，模块职责明确，尽量保持正交。
- Debug 只给“预期 vs 实际 + 最小复现”；测试可交给 AI，断言需人工审。
- 频繁“新会话”：代码一多就切新对话，重新注入固定上下文。

## 6. 快速提示词选型（照抄即可用）
- 开局：系统提示词 v10/v8 + `项目上下文文档生成.md`
- 需求澄清与方案：`智能需求理解与研发导航引擎.md`
- 计划执行：`标准化流程.md`，必要时加 `系统架构可视化`（生成 mermaid）
- 复盘与移交：更新 `progress.md` / `architecture.md`，按需要生成交付说明

## 7. 验收清单（新/老项目通用）
- [ ] 系统提示词已注入并设置 Always 读取上下文文件
- [ ] 上下文文档最新，缺失项明确标“暂无信息”
- [ ] 当前任务有拆分计划，且每步有验证方式
- [ ] `progress.md`、`architecture.md` 已更新
- [ ] 变更已测试并以小步提交
