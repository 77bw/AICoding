## 一、核心理念

### 1.1 Vibe Coding 是什么

Vibe Coding = **规划驱动 + 上下文固定 + AI 结对执行**

核心公式：
```
优质上下文 + 精准提示词 + 分步执行 = 高质量代码产出
```

### 1.2 三大核心原则

| 原则 | 说明 | 反模式 |
|------|------|--------|
| 规划先行 | 先让 AI 给方案，确认后再执行 | 直接让 AI 写代码 |
| 上下文即生命 | 垃圾进，垃圾出 | 不提供项目背景就开干 |
| 小步验证 | 一次只做一件事，做完验证 | 一口气让 AI 实现整个功能 |

### 1.3 道·法·术·器

**道**（心法）：
- 凡是 AI 能做的，就不要人工做
- 目的主导：开发过程中的一切动作围绕"目的"展开
- 先结构，后代码，一定要规划好框架
- 奥卡姆剃刀：如无必要，勿增代码

**法**（方法）：
- 一句话目标 + 非目标
- 按职责拆模块
- 接口先行，实现后补
- 文档即上下文，不是事后补

**术**（技巧）：
- 明确写清：能改什么，不能改什么
- Debug 只给：预期 vs 实际 + 最小复现
- 代码一多就切会话

**器**（工具）：
- Claude Code / Codex CLI：终端 AI 编程
- Cursor / Windsurf：IDE 集成
- Augment：上下文引擎
- 本仓库：提示词、技能、文档资源库

## 三、新项目：从零开始

### 3.1 第一步：创建项目骨架

```
your-project/
├── .cursorrules          # ← 系统提示词（Cursor 用户）
├── CLAUDE.md             # ← 项目上下文 + 系统提示词（通用）
├── AGENTS.md             # ← 项目上下文（Codex 用户）
├── progress.md           # ← 进度记录
├── docs/
│   ├── prd.md            # 产品需求文档
│   └── tech-stack.md     # 技术栈说明
├── src/                  # 源代码
└── tests/                # 测试
```

### 3.2 第二步：配置系统提示词

**方式一：Cursor IDE 用户**

1. 复制系统提示词到 `.cursorrules`：
```bash
# 从本仓库复制（根据你的路径调整）
cp vibe-coding-cn/i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md .cursorrules
```

2. 创建 `CLAUDE.md` 存放项目上下文（见下一步）

**方式二：Claude Code / Codex CLI 用户**

直接将系统提示词放在 `CLAUDE.md` 或 `AGENTS.md` 顶部，然后添加项目上下文。

### 3.3 第三步：生成项目上下文

使用「项目上下文文档生成」提示词，让 AI 帮你生成初版 CLAUDE.md：

```
[粘贴 coding_prompts/项目上下文文档生成.md 的内容]

你需要处理的是：我正在开发一个 [项目描述]，技术栈是 [技术栈]
```

生成后保存为 `CLAUDE.md`，并人工审核补充。

### 3.4 第四步：创建进度文件

创建 `progress.md`：

```markdown
# 开发进度记录

## 当前状态
- 项目阶段：初始化
- 最后更新：2024-12-21

## 里程碑

### 2024-12-21 项目初始化
- [x] 创建项目骨架
- [x] 配置 .cursorrules 和 CLAUDE.md
- [ ] 实现核心功能 A
- [ ] 实现核心功能 B

## 待办事项
1. ...
2. ...
```

### 3.5 第五步：开始 Vibe Coding

启动第一个任务：

```
阅读 CLAUDE.md 和 progress.md，我接下来要做：[任务描述]

请先给出实施计划，不要直接写代码。我确认后再执行。
```

**关键**：始终先让 AI 给方案，确认后再执行！

## 四、老项目：快速接入

### 4.1 第一步：盘点现状

- 梳理现有文档（README、架构图等）
- 列出核心目录和关键文件
- 明确本次改造范围和非目标

### 4.2 第二步：添加配置文件

```bash
# Cursor 用户
cp vibe-coding-cn/i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md .cursorrules

# 创建上下文文件
touch CLAUDE.md progress.md
```

### 4.3 第三步：让 AI 帮你生成上下文

在你的 AI 工具中输入：

```
请阅读当前项目的目录结构和主要文件，然后生成一份完整的项目上下文文档（CLAUDE.md），包含：
- 项目概要（名称、目标、约束）
- 技术栈
- 目录结构与关键文件职责
- 当前进度
- 开发约定

对于无法确定的信息，标注为"暂无信息"
```

### 4.4 第四步：人工审核补充

生成的 CLAUDE.md 需要人工审核：
- 补充 AI 无法推断的业务逻辑
- 明确约束和非目标
- 添加特殊注意事项

### 4.5 第五步：记录项目现状

在 `progress.md` 中记录：

```markdown
# 开发进度记录

## 项目现状（接入日期：2024-12-21）
- 已有模块：[列出现有模块]
- 本次改造目标：[明确目标]
- 暂不涉及：[非目标范围]

## 后续里程碑
- [ ] 任务 1
- [ ] 任务 2
```

## 五、日常开发工作流

### 5.1 标准工作流程图

```
┌─────────────────────────────────────────────────────────────┐
│                    开启任务                                  │
│  阅读上下文 → 描述需求 → AI 给方案 → 确认方案               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    执行开发                                  │
│  小步实现 → 验证测试 → 确认无误 → Git 提交                   │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│                    完成任务                                  │
│  更新 CLAUDE.md → 更新 progress.md → 切换新会话             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 开启新任务

```
阅读 CLAUDE.md 和 progress.md，我接下来要做：[任务描述]

请先给出实施计划，包括：
1. 需要修改哪些文件
2. 具体修改步骤
3. 如何验证

不要直接写代码，等我确认后再执行。
```

### 5.3 需求不清晰时

使用「智能需求理解与研发导航引擎」：

```
[粘贴 智能需求理解与研发导航引擎.md 的内容]

你需要处理的是：[你的模糊需求]
```

AI 会输出：
- 需求理解与意图识别
- 关键词·概念·隐性知识
- 技术路径·开源项目·参考资料
- 专家范式·高层洞见与建议

### 5.4 Debug 时

标准 Debug 模板：

```
遇到问题，请帮我分析：

**预期行为**：
[应该发生什么]

**实际行为**：
[实际发生了什么]

**错误信息**：
[错误日志/堆栈]

**最小复现步骤**：
1. ...
2. ...
3. ...

**相关代码位置**：
[文件路径和行号]
```

### 5.5 完成任务后

```
任务已完成，请帮我：
1. 总结本次改动的要点
2. 更新 CLAUDE.md 中受影响的部分（给出具体内容）
3. 生成 progress.md 的更新条目
```

### 5.6 切换新会话

当对话过长或上下文混乱时：

1. 在 Claude Code 中：`/clear` 或 `/compact`
2. 在 Cursor 中：开启新 Chat
3. 新会话第一句话：

```
阅读 CLAUDE.md 和 progress.md，了解项目现状。
上次进度已记录在 progress.md，请从 [指定任务] 继续。
```

## 六、进阶技巧

### 6.1 深度思考模式

在 Claude Code 中触发深度思考：
- `think` < `think hard` < `think harder` < `ultrathink`

使用场景：复杂架构决策、疑难 Bug 分析、重构方案设计

### 6.2 使用 Skills

本仓库提供了多个专业技能包，使用方法：

```
阅读 [技能路径]/SKILL.md，然后帮我 [任务描述]
```

示例：
```
阅读 skills/postgresql/SKILL.md，然后帮我设计一个用户表，包含基本信息和登录日志
```

### 6.3 生成新的 Skills

使用元技能 `claude-skills`：

```
阅读 skills/claude-skills/SKILL.md，帮我为 [某个库/框架] 生成一个 SKILL
```

### 6.4 多 AI 协作

推荐工作流：
1. **Claude Opus 4.5**：复杂架构设计、核心代码实现
2. **Gemini**：文档整理、测试用例生成
3. **另一个 AI**：Code Review

```
我用 [AI 1] 写了以下代码，请帮我 review：
[代码]

重点关注：
- 是否有潜在 Bug
- 是否符合最佳实践
- 是否有安全隐患
```

### 6.5 文档同步策略

**强制规则**：每次架构变更必须同步更新 CLAUDE.md

包括：
- 新增/删除/移动文件
- 模块职责变化
- 接口变更
- 依赖变化

## 七、常见问题

### Q1：系统提示词放 .cursorrules 还是 CLAUDE.md？

- **Cursor 用户**：系统提示词放 `.cursorrules`，项目上下文放 `CLAUDE.md`
- **Claude Code / Codex 用户**：都放 `CLAUDE.md`（系统提示词在前，项目上下文在后）

### Q2：AI 不遵守规则怎么办？

1. 检查配置文件是否在项目根目录
2. 在提示词开头强调："请先阅读 CLAUDE.md"
3. 切换新会话重新注入上下文
4. 使用更强的约束词："必须"、"禁止"、"强制"

### Q3：项目太大 AI 读不完怎么办？

1. CLAUDE.md 只写核心模块的一句话职责，不要全量代码
2. 按模块拆分任务，每次只让 AI 关注一个模块
3. 使用 `@file` 语法（Cursor）指定阅读特定文件
4. 使用 Augment 等上下文引擎

### Q4：如何让 AI 记住之前的决策？

1. 每次重要决策都记录在 CLAUDE.md 的"架构决策"部分
2. 关键代码添加注释说明设计意图
3. 新会话时先让 AI 阅读上下文文档

### Q5：如何处理多人协作？

1. CLAUDE.md 和 progress.md 提交到 Git
2. 每次开始工作先 pull 最新文档
3. 完成工作后先更新文档再 push
4. 使用统一的系统提示词版本

### Q6：系统提示词选哪个版本？

| 版本 | 适用场景 |
|------|----------|
| v10 | 正式项目、复杂架构、长期维护 |
| v8 | 快速原型、小型项目、简单任务 |
