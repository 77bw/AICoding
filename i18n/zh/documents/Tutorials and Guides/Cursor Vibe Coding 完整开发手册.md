## 一、Cursor 核心概念

### 1.1 Cursor 的三种 AI 交互模式

| 模式 | 快捷键 | 适用场景 | 特点 |
|------|--------|----------|------|
| **Chat** | `Cmd/Ctrl + L` | 问答、分析、规划 | 对话式，可引用文件 |
| **Composer** | `Cmd/Ctrl + I` | 多文件编辑、重构 | 可同时修改多个文件 |
| **Inline Edit** | `Cmd/Ctrl + K` | 局部代码修改 | 选中代码后快速编辑 |

### 1.2 Cursor 的规则注入层级

```
┌─────────────────────────────────────────────────────────────┐
│  Level 1: Settings → Rules for AI                          │
│  作用域：全局（所有项目）                                    │
│  内容：个人偏好、通用编码规范                                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Level 2: 项目根目录 .cursorrules                           │
│  作用域：当前项目                                            │
│  内容：系统提示词、AI 行为规范                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Level 3: 项目根目录 CLAUDE.md / README.md                  │
│  作用域：当前项目                                            │
│  内容：项目上下文、架构说明、开发约定                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Level 4: 对话中 @file 引用                                 │
│  作用域：当前对话                                            │
│  内容：特定文件的上下文                                      │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 Vibe Coding 在 Cursor 中的核心公式

```
.cursorrules（系统提示词）
    + CLAUDE.md（项目上下文）
    + progress.md（进度记录）
    + @file 引用（临时上下文）
    = 高质量 AI 辅助开发
```

## 二、环境配置

### 2.1 全局设置（Settings → Rules for AI）

打开 Cursor 设置，找到 `Rules for AI`，添加以下内容：

```markdown
# 全局规则

## 语言偏好
- 代码注释使用中文
- 变量名、函数名使用英文
- 回复使用中文

## 编码规范
- 遵循项目现有代码风格
- 优先使用项目已有的工具函数
- 不随意添加新依赖

## 交互规范
- 修改代码前先说明要改什么、为什么改
- 不确定的地方先问我
- 给出方案后等我确认再执行
```

### 2.2 Cursor 推荐设置

在 `Settings` 中配置：

| 设置项 | 推荐值 | 说明 |
|--------|--------|------|
| `Cursor Tab` | 开启 | AI 代码补全 |
| `Copilot++` | 开启 | 增强补全 |
| `Auto-import` | 开启 | 自动导入 |
| `Max Tokens` | 4000+ | 更长的上下文 |

### 2.3 模型选择建议

| 任务类型 | 推荐模型 | 原因 |
|----------|----------|------|
| 复杂架构设计 | Claude Opus 4.5 / GPT-4 | 推理能力强 |
| 日常编码 | Claude Sonnet / GPT-4o | 性价比高 |
| 快速原型 | Claude Haiku / GPT-4o-mini | 速度快 |

## 三、三大核心文件

### 3.1 文件结构总览

```
your-project/
├── .cursorrules          # ← 系统提示词（Cursor 专用）
├── CLAUDE.md             # ← 项目上下文（通用）
├── progress.md           # ← 进度记录
└── ...
```

### 3.2 .cursorrules 详解

**作用**：定义 AI 的行为规范、思维模式、输出风格

**完整模板**（从本仓库复制）：

```bash
# 复制系统提示词 v10（推荐）
cp vibe-coding-cn/i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md .cursorrules

# 或复制简洁版 v8
cp vibe-coding-cn/i18n/zh/prompts/system_prompts/CLAUDE.md/8/CLAUDE.md .cursorrules
```

**精简版模板**（如果不想用完整版）：

```markdown
# 系统规则

## 角色定位
你是一位专业的软件工程师，擅长 [技术栈]。

## 工作模式
1. 修改代码前先说明计划
2. 一次只做一件事
3. 每次修改后说明改了什么

## 代码规范
- 函数不超过 20 行
- 使用有意义的命名
- 添加必要的注释

## 禁止事项
- 不要删除现有注释
- 不要改动不相关的代码
- 不要引入新依赖（除非我明确要求）

## 输出格式
- 使用中文回复
- 代码使用 markdown 代码块
- 复杂逻辑用列表说明
```

### 3.3 CLAUDE.md 详解

**作用**：存储项目上下文、架构信息、开发约定

**完整模板**：

```markdown
# 项目上下文

## 项目概要
- **项目名称**：[项目名]
- **项目目标**：[一句话描述]
- **非目标/约束**：[不做什么]

## 技术栈
- **语言**：Python 3.11
- **框架**：FastAPI
- **数据库**：PostgreSQL
- **缓存**：Redis
- **部署**：Docker

## 目录结构
```text
src/
├── api/              # API 接口层
│   ├── v1/           # 版本 1 接口
│   └── deps.py       # 依赖注入
├── core/             # 核心业务逻辑
│   ├── services/     # 业务服务
│   └── models/       # 数据模型
├── data/             # 数据访问层
│   └── repository/   # 数据仓库
└── utils/            # 工具函数
```

## 关键文件职责
| 文件 | 职责 |
|------|------|
| `src/main.py` | 程序入口，启动 FastAPI |
| `src/core/config.py` | 配置管理，读取环境变量 |
| `src/api/deps.py` | 依赖注入，数据库连接 |

## 架构决策记录
1. **2024-12-20**：选择 FastAPI 而非 Flask，因为需要异步支持
2. **2024-12-21**：使用 Repository 模式隔离数据访问

## 开发约定
1. 写代码前必须阅读本文件
2. 变更架构后必须更新本文件
3. 每个模块完成后更新 progress.md
4. API 必须有类型注解和文档字符串

## 当前重点
- 正在开发：用户认证模块
- 下一步：权限管理模块
- 已完成：项目骨架、数据库连接
```

## 四、新项目初始化

### 4.1 快速初始化脚本

创建 `init-vibe-coding.sh`：

```bash
#!/bin/bash

# 创建目录结构
mkdir -p src/{api,core,data,utils} tests docs

# 复制系统提示词
cp /path/to/vibe-coding-cn/i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md .cursorrules

# 创建 CLAUDE.md
cat > CLAUDE.md << 'EOF'
# 项目上下文

## 项目概要
- **项目名称**：[待填写]
- **项目目标**：[待填写]
- **非目标/约束**：[待填写]

## 技术栈
[待填写]

## 目录结构
```text
src/
├── api/
├── core/
├── data/
└── utils/
```

## 开发约定
1. 写代码前必须阅读本文件
2. 变更架构后必须更新本文件
EOF

# 创建 progress.md
cat > progress.md << 'EOF'
# 开发进度记录

## 当前状态
- **项目阶段**：初始化
- **最后更新**：$(date +%Y-%m-%d)

## 里程碑

### $(date +%Y-%m-%d) 项目初始化
- [x] 创建项目骨架
- [x] 配置 Vibe Coding 环境
- [ ] [下一步任务]
EOF

echo "Vibe Coding 环境初始化完成！"
```

### 4.2 分步初始化流程

**第一步**：创建项目并打开 Cursor

```bash
mkdir my-project && cd my-project
cursor .
```

**第二步**：创建 .cursorrules

在 Cursor Chat 中输入：

```
帮我创建一个 .cursorrules 文件，包含以下内容：
1. 你是一个专业的 [技术栈] 开发者
2. 代码规范：[你的规范]
3. 输出规范：中文回复，代码前说明修改计划
```

或直接从本仓库复制：`i18n/zh/prompts/system_prompts/CLAUDE.md/10/CLAUDE.md`

**第三步**：生成 CLAUDE.md

使用「项目上下文文档生成」提示词：

```
我正在开发一个 [项目描述]。

技术栈：
- 语言：[语言]
- 框架：[框架]
- 数据库：[数据库]

请帮我生成一份完整的 CLAUDE.md 项目上下文文档，包含：
- 项目概要
- 技术栈
- 目录结构建议
- 开发约定
```

**第四步**：创建 progress.md

```
帮我创建一个 progress.md 文件，用于记录开发进度。
当前状态：项目刚初始化
接下来要做：[你的计划]
```

**第五步**：验证配置

```
请阅读 .cursorrules、CLAUDE.md 和 progress.md，总结你对本项目的理解。
```

如果 AI 能正确复述，说明配置成功。

## 六、日常开发工作流

### 6.1 标准工作流

```
┌──────────────────────────────────────────────────────────────┐
│  1. 开始工作                                                  │
│  打开 Cursor → Chat → 阅读上下文                              │
└──────────────────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────────────────┐
│  2. 描述任务                                                  │
│  清晰描述要做什么 → AI 给出方案 → 确认方案                    │
└──────────────────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────────────────┐
│  3. 执行开发                                                  │
│  Composer 多文件编辑 / Inline Edit 局部修改                   │
└──────────────────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────────────────┐
│  4. 验证测试                                                  │
│  运行代码 → 检查结果 → 修复问题                               │
└──────────────────────────────────────────────────────────────┘
         │
         ▼
┌──────────────────────────────────────────────────────────────┐
│  5. 完成收尾                                                  │
│  更新 CLAUDE.md → 更新 progress.md → Git 提交                 │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 Chat 模式工作流

**开始任务**：

```
阅读 CLAUDE.md 和 progress.md，我接下来要做：

[任务描述]

请先给出实施方案，包括：
1. 需要修改/新建哪些文件
2. 具体修改步骤
3. 如何验证

不要直接写代码，等我确认后再执行。
```

**确认后执行**：

```
方案确认，请开始实现。一次只改一个文件，改完告诉我。
```

**验证通过后**：

```
功能验证通过。请帮我：
1. 总结本次改动
2. 更新 CLAUDE.md（如有架构变更）
3. 更新 progress.md 的进度
```

### 6.3 Composer 模式工作流

**适用场景**：需要同时修改多个文件

**操作步骤**：

1. 按 `Cmd/Ctrl + I` 打开 Composer
2. 描述需求：

```
我需要添加一个用户登录功能：

1. 在 src/api/v1/auth.py 添加登录接口
2. 在 src/core/services/auth.py 添加登录逻辑
3. 在 src/core/models/user.py 添加用户模型
4. 在 tests/test_auth.py 添加测试用例

请按照 CLAUDE.md 中的代码规范实现。
```

3. 检查 AI 生成的修改预览
4. 点击 Accept 应用修改
5. 运行测试验证

### 6.4 Inline Edit 工作流

**适用场景**：修改特定代码片段

**操作步骤**：

1. 选中要修改的代码
2. 按 `Cmd/Ctrl + K`
3. 描述修改需求：

```
将这个函数改为异步函数，添加错误处理
```

4. 检查预览，按 Tab 接受或 Esc 拒绝

### 6.5 @file 引用技巧

在 Chat 中使用 `@` 引用特定文件：

```
@src/api/v1/user.py @src/core/models/user.py

请分析这两个文件的关系，并优化 API 的参数校验逻辑。
```

**常用引用模式**：

| 引用方式 | 说明 |
|----------|------|
| `@filename` | 引用单个文件 |
| `@folder/` | 引用整个文件夹 |
| `@Codebase` | 搜索整个代码库 |
| `@Docs` | 引用项目文档 |
| `@Web` | 搜索网络 |

## 七、Cursor 高级功能

### 7.1 快捷键速查表

| 功能 | Mac | Windows/Linux |

| 打开 Composer | `Cmd + I` | `Ctrl + I` |

| 新建 Chat | `Cmd + Shift + L` | `Ctrl + Shift + L` |
| 清空 Chat | `/clear` | `/clear` |
| 切换模型 | 点击模型名称 | 点击模型名称 |

### 7.2 Agent 模式

**开启方式**：在 Chat 中点击模式切换，选择 "Agent"

**适用场景**：
- 需要 AI 自主探索代码库
- 复杂的重构任务
- 多步骤的开发任务

**使用示例**：

```
[Agent 模式]

请帮我重构用户模块：
1. 分析当前代码结构
2. 找出需要优化的地方
3. 给出重构方案
4. 实施重构

过程中如有不确定的地方请问我。
```

### 7.3 代码解释功能

选中代码后，右键选择 "Explain Code" 或使用：

```
@selected 解释这段代码的作用和实现原理
```

### 7.4 代码生成快捷方式

在代码中输入注释，然后按 `Cmd/Ctrl + K`：

```python
# 实现一个函数：接收用户ID，返回用户信息，如果不存在返回 None
# [按 Cmd+K 触发生成]
```

### 7.5 多文件搜索与重构

使用 `@Codebase` 进行代码库搜索：

```
@Codebase 找出所有使用了 deprecated_function 的地方，并告诉我如何替换
```

## 十、故障排除

### 10.1 AI 不遵守 .cursorrules

**症状**：AI 输出不符合规则定义

**解决方案**：
1. 检查 `.cursorrules` 是否在项目根目录
2. 重启 Cursor
3. 在提示词中显式强调：

```
严格按照 .cursorrules 中的规范执行：
[你的请求]
```

### 10.2 AI 忘记项目上下文

**症状**：AI 对项目一无所知

**解决方案**：
1. 开启新 Chat 时先注入上下文：

```
阅读 CLAUDE.md 和 progress.md，了解项目现状后回复"已了解"
```

2. 使用 `@CLAUDE.md` 强制引用

### 10.3 Composer 修改不准确

**症状**：Composer 改错文件或改错位置

**解决方案**：
1. 更精确地描述文件路径
2. 使用 `@file` 明确指定文件
3. 拆分为小任务，一次只改一个文件

### 10.4 Chat 上下文太长

**症状**：响应变慢，AI 开始忘记前面的内容

**解决方案**：
1. 使用 `/clear` 清空对话
2. 重新注入核心上下文
3. 确保 progress.md 记录了关键进度

### 10.5 代码生成质量差

**症状**：生成的代码不符合项目风格

**解决方案**：
1. 在 `.cursorrules` 中添加更详细的代码规范
2. 使用 `@file` 引用现有代码作为示例：

```
参考 @src/existing_module.py 的代码风格，实现 [新功能]
```

### 10.6 常见错误信息

| 错误 | 原因 | 解决方案 |
|------|------|----------|
| "I don't have access to..." | 文件未被索引 | 重新打开项目或手动引用 |
| "The file is too large" | 文件超过限制 | 引用具体函数或片段 |
| "Rate limit exceeded" | API 请求过多 | 等待后重试 |
